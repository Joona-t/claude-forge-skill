---
description: Meta-skill that creates new CLI skills on the fly. Generates properly formatted .md skill files with template skeletons.
argument: name scope "purpose" [--template audit|build|research|fix|swarm-dispatch|utility]
---

Generate a new Claude Code skill file from a template. The inception — a skill that creates skills.

## Arguments

Parse `$ARGUMENTS` as:
- **name** — kebab-case skill name (required)
- **scope** — `global` or `project` (required)
- **purpose** — quoted description of what the skill does (required)
- **--template** — optional: `audit`, `build`, `research`, `fix`, `swarm-dispatch`, `utility`

If `--template` is omitted, infer from purpose keywords:
- "check", "scan", "verify", "audit", "validate" → `audit`
- "create", "build", "generate", "construct", "scaffold" → `build`
- "investigate", "analyze", "explore", "research", "compare" → `research`
- "fix", "repair", "patch", "hotfix" → `fix`
- "swarm", "deploy agents", "orchestrate", "parallel agents" → `swarm-dispatch`
- Otherwise → `utility`

## Step 1: Validate

1. Confirm name is kebab-case (lowercase, hyphens only, no spaces/underscores)
2. Set target directory:
   - `global` → `~/.claude/commands/`
   - `project` → `.claude/commands/` (relative to cwd)
3. Check BOTH `~/.claude/commands/` AND `.claude/commands/` for existing `{name}.md` — if found, **WARN** the user and stop. Do not overwrite.
4. Count files in target directory. Warn if >25 global or >10 project skills exist (but don't block).

## Step 2: Select Template

Match the `--template` flag or inferred template from Step 1. Use the corresponding skeleton from the Templates section below.

## Step 3: Generate

Fill the template skeleton:
- Replace `{{NAME}}` with the skill name
- Replace `{{PURPOSE}}` with the purpose description
- Replace `{{DATE}}` with today's date (YYYY-MM-DD)
- Keep `$ARGUMENTS` placeholder intact for runtime argument passing
- Do NOT hardcode file paths — use relative references or `~/` notation
- Include a memory load step for `audit` and `build` templates

## Step 4: Write

1. Use the Write tool to create the .md file at the target path
2. Read the file back to confirm it was written correctly
3. Verify: valid markdown, `$ARGUMENTS` present if the template uses it, no hardcoded absolute paths outside `~/`

## Step 5: Register

Append an entry to `~/.claude/docs/memory/tool-changelog.md`:

```markdown
---

## Skill: /{{NAME}}
- **Location:** `{{target_path}}`
- **Version:** 1.0 ({{DATE}})
- **What it does:** {{PURPOSE}}
- **Generated by:** /forge-skill (template: {{template_type}})
- **Known gaps:** Newly generated — not yet tested end-to-end.
- **Improvement backlog:** Test and refine based on first real usage.
```

Tell the user the skill is ready: `/{{NAME}}` is now available.

## Templates

### audit

```markdown
---
description: {{PURPOSE}}
argument: target path or extension name (required)
---

Audit: {{PURPOSE}}

## Step 0: Load Memory

Read these files before starting:
- `~/.claude/docs/memory/known-issues.md`
- `~/.claude/docs/memory/color-decisions.md` (if UI-related)
- `~/.claude/docs/memory/debugging.md`

## Step 1: Identify Target

Parse `$ARGUMENTS` to determine what to audit. Verify the target exists.

## Step 2: Domain Checks

Run the domain-specific checks for this audit:
- [ ] Check 1 — describe what to verify
- [ ] Check 2 — describe what to verify
- [ ] Check 3 — describe what to verify

Customize these checks based on the audit's purpose.

## Step 3: Summary Table

Present findings as a table:

| Check | Status | Details |
|-------|--------|---------|
| Check 1 | PASS/FAIL | ... |

## Step 4: Knowledge Base Update

If new issues were found:
- Add to `~/.claude/docs/memory/known-issues.md` if novel
- Flag recurrences of existing KI entries
- Suggest `/postmortem` if significant findings
```

### build

```markdown
---
description: {{PURPOSE}}
argument: what to build (required)
---

Build: {{PURPOSE}}

## Step 1: Load Context

Read relevant context files:
- `~/.claude/docs/memory/known-issues.md`
- `~/.claude/docs/memory/debugging.md`
- Any domain-specific docs relevant to this build

## Step 2: Plan

Outline what will be built based on `$ARGUMENTS`. Present the plan and wait for confirmation before proceeding.

## Step 3: Write

Implement the planned components. Follow existing patterns and conventions.

## Step 4: Verify

Run relevant verification:
- Check output exists and is valid
- Run any applicable audits
- Confirm no regressions

## Step 5: Postmortem

Suggest running `/postmortem` to capture learnings.
```

### research

```markdown
---
description: {{PURPOSE}}

---

## Step 1: Dry Run

Parse `$ARGUMENTS`. Outline research scope and approach. Confirm with user before deep-diving.

## Step 3: Synthesize

Summarize key findings, trade-offs, and recommendations.
```

```markdown
---
description: {{PURPOSE}}

Quick fix: {{PURPOSE}}
```

### swarm-dispatch

```markdown
---
description: {{PURPOSE}}
argument: target and parameters (required)
---

Swarm dispatch: {{PURPOSE}}

Read `AGENT-SWARM.md` for the full agent roster and execution protocol.

## Agents to Deploy

Select the relevant agents for `$ARGUMENTS`:
- Lead: [which council lead owns this domain]
- Specialists: [which specialists to deploy]

## Execution

Follow the 4-wave pattern from AGENT-SWARM.md:
1. **Wave 1 (Recon):** Deploy selected specialists in parallel
2. **Wave 2 (Analysis):** Lead analyzes specialist reports
3. **Wave 3 (Gate):** Linus GO/NO-GO
4. **Wave 4 (Action):** Apply fixes if needed

Output reports to `_reports/`.
```

## Swarm Integration

An agent should log `FORGE CANDIDATE: <name> -- <purpose>` to session-memory.md when ALL conditions are met:
- The workflow has been done manually before (or addresses a known-issues.md pattern)
- It's self-contained (doesn't depend on in-flight swarm state)
- It takes >3 steps to perform manually
- Reuse scope is clear (global if 3+ extensions benefit, project if extension-specific)

Actual forging happens in **Wave 4** (post-build) or during `/postmortem` — never during active Wave 1-3 execution.
